# H100 Speed Audit

Date: 2026-04-30

## What Changed

The first RunPod H100 launch proved the bundle/data path was healthy, but also
exposed three avoidable speed costs in the paid profile:

- The initial 1xH100 smoke inherited an 8xH100 batch shape and failed before
  training. The runner now overrides 1x runs to `TRAIN_BATCH_TOKENS=131072`,
  `VAL_BATCH_SIZE=131072`, and `GRAD_ACCUM_STEPS=1`.
- The first real H100 row had `TRAIN_FUSED_QKV=0`. Enabling fused QKV preserved
  Q/K/V capacity while reducing projection launches and train-time q8
  materializations.
- `TRAIN_ABORT_ON_NONFINITE=1` was inherited from the local safety profile. That
  scans gradients and parameters every step. After the row family smoked cleanly,
  the H100 scout profile disables those scans.

The H100 profile now also uses `WARMUP_STEPS=0` while compile is disabled,
`POST_STEP_ZERO_GRAD=0` because gradients are cleared at the start of each step,
and `TRAIN_LOG_EVERY=250`.

Two correctness fixes landed after the first H100 scout:

- `TRAIN_QUANT_EMBEDDINGS=1` now fake-quantizes the tied token embedding during
  training. Before this, linears were trained through q8 but `tok_emb.weight`
  was only quantized at export, creating a large final roundtrip gap.
- HRC candidate helpers now force `MODEL_FAMILY=hrc`. The first H100 round used
  HRC-shaped names but actually trained the baseline path because the inherited
  route helper only set `NUM_UNIQUE_BLOCKS` / `EFFECTIVE_DEPTH`.

## Measured Step Speed

Initial speed numbers were measured on the accidentally-baseline
`h100_capfill_i3l5r5_d640e896_q8_polar`, 1xH100, BF16 autocast, fp32
params/Muon, q8 train-time forward, `TRAIN_BATCH_TOKENS=131072`.

| Profile | Step Speed |
| --- | ---: |
| 1x batch fix, unfused QKV, nonfinite scans on | ~314 ms/step |
| fused QKV | ~298 ms/step |
| fused QKV + no nonfinite scans + no post-step zero | ~271 ms/step |

Net win: about 14% more optimizer steps in the same training wallclock, without
changing the model math.

## Batch Sweep

Increasing single-H100 batch size uses more memory, but did not improve proxy
quality at equal 35-second wallclock. Bigger batches gave only a small
tokens/sec gain while reducing optimizer updates sharply.

| Train Batch Tokens | Steps in 35s | Step Speed | Approx Tokens/s | Proxy BPB |
| ---: | ---: | ---: | ---: | ---: |
| 65,536 | 137 | 255.77 ms | 256k | 2.1439 |
| 81,920 | 113 | 312.09 ms | 263k | 2.1535 |
| 98,304 | 96 | 367.86 ms | 267k | 2.1645 |
| 114,688 | 83 | 424.20 ms | 270k | 2.1751 |

Conclusion: do not chase 80GB memory occupancy for its own sake on the 1xH100
scout. The best short-run signal is still the 65k-token per-rank batch because
this family appears update-count limited, not memory-capacity limited.

## Next Speed Levers

- Test `LOSS_FP32=0` only as a speed/quality probe; it changes numeric behavior
  and should not become default without export-BPB evidence.
- Test BF16 trainable params separately from the official-quality scout. The
  local fp16-param path collapsed export BPB, but BF16 on H100 may be less bad.
- Consider a fused dense qN materialization CUDA kernel for q8/q6 fake-quant
  weights. The current train-time quant path still builds quantized weights from
  several PyTorch ops per unique linear per forward.
- Keep `torch.compile` disabled unless a separate smoke proves compile time is
  repaid within a 10-minute run.

## H100 Cap-Fit Preflight Gate

Important process fix from the e1536/e1728 over-cap results: every new
near-cap H100 group should run a tiny export preflight before the real proxy.
The preflight uses one training step plus final artifact export. It is not a
perfect final-size predictor because training changes weight entropy, but it is
good enough to reject obvious over-cap shapes before spending a 6-10 minute
training slot.

Code changes:

- `scripts/run_h100_novel_matrix.py` now accepts `--iterations` and
  `--warmdown-iters`, so we can run `ITERATIONS=1` artifact exports.
- `scripts/run_h100_group_preflight_then_proxy.sh` lists a candidate group,
  runs each row through a 1-step export, rejects rows with
  `artifact_total_bytes > 16,000,000`, then only proxy-trains accepted rows.
- The wrapper can also skip the final 600s rerun if the best proxy is not
  promising via `FINAL_MAX_PROXY_BPB`.
- `scripts/guard_h100_proxy_final.sh` was added as a live guard for groups that
  were already launched before the preflight wrapper existed. It watches the
  proxy runner and stops the automatic final rerun unless the best proxy clears
  a threshold.

## 1xH100 Versus 8xH100 Strategy

The 8xH100 run is not expected to give eight times more optimizer steps. With
the current runner, 1xH100 uses `65,536` train tokens per step, while 8xH100
uses the official-style `524,288` global train tokens per step, roughly the same
per-rank microbatch. Step time should be in the same broad range plus
distributed overhead, but each step sees about 8x more data.

Implications:

- Use 1xH100 for architecture ranking, cap sanity, and obvious speed failures.
- Use 8xH100 for confirmation under official-style data throughput and for
  tuning batch-sensitive training knobs.
- Do not assume the 1x winner is automatically the 8x winner. Larger global
  batch changes gradient noise and may need LR/min-LR/warmdown retuning.
- The first 8x matrix should be small: two or three best architectures crossed
  with one or two training-schedule variants, not a broad architecture sweep.
